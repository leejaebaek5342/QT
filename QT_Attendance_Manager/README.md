# Qt로 출석 관리 프로그램 만들기

QWidget, QStackedWidget, QTableWidget, QComboBox 등을 활용해  
선생 로그인, 학생 등록, 출석 저장, 학생 조회가 가능한 출석 관리 프로그램을 만들어보았다.

## 이 프로그램은 다음과 같은 흐름으로 동작한다

- 메인 화면에서 Teacher / Student 선택
- Teacher 회원가입
- Teacher 로그인
- 학생 등록
- 학생 목록 확인
- 날짜별 출석 상태 저장
- 학생별 출석 조회
- Student 화면에서 이름과 학번이 일치하는 학생의 출석 조회

## 프로그램 구성

이번 프로그램은 크게 Teacher 화면과 Student 화면으로 나누어 구성했다.  
Teacher 화면에서는 회원가입, 로그인, 학생 등록, 학생 목록 확인, 출석 저장, 학생별 조회를 할 수 있고,  
Student 화면에서는 이름과 학번을 입력해 일치하는 학생의 출석 기록을 확인할 수 있도록 만들었다.

이번 글에서는 전체 코드를 모두 다루기보다는, 구현 방식과 문제 해결 과정, 그리고 느낀점을 중심으로 정리해보려고 한다.

## 구현 방식

화면 전환은 `QStackedWidget`을 이용해 Teacher 화면과 Student 화면을 나누어 처리했고,  
선생 회원가입은 따로 위젯을 만들어 새로운 창이 생성되도록 만들었다.  
회원가입은 별도의 창이 뜨는 방식이 더 자연스럽다고 생각했기 때문이다.

학생 목록은 `QTableWidget`에 출력하고, 출석 상태는 각 행마다 `QComboBox`를 배치해 선택할 수 있도록 구성했다.

데이터 저장은 데이터베이스 대신 텍스트 파일 방식을 사용했다.  
교사 계정 정보는 `teachers.txt`에 저장하고, 학생 목록과 출석 정보는 로그인한 선생님의 ID를 기준으로  
`studentslist.txt`, `attendance.txt` 형식으로 분리해 저장하도록 구현했다.

세부 기능 설명은 GitHub에 올린 코드에서 각 함수 위 주석으로 함께 정리해두었다.

## 문제 해결

처음에는 학생 목록과 출석 정보를 공용 파일 하나로 관리했기 때문에  
선생님이 바뀌어도 이전 선생님의 학생 목록이 그대로 보이는 문제가 있었다.

이 문제를 해결하기 위해 선생님 ID를 기준으로 파일명을 다르게 만들어  
학생 목록과 출석 정보를 선생님별로 분리해서 저장하도록 수정했다.

이를 위해 로그인한 선생님의 ID를 기준으로  
`studentslist_<teacherId>.txt`, `attendance_<teacherId>.txt` 형식의 파일명을 사용하도록 구현했다.

이 부분을 수정하고 나서야 로그인한 선생님마다 서로 다른 학생 목록과 출석 정보를 관리할 수 있게 되었다.

또 하나 고민했던 부분은 함수의 반환값 처리 방식이었다.  
처음에는 중복 검사나 학생 정보 확인 같은 함수들을 `bool`로 처리할 수도 있다고 생각했지만,  
이 경우 "파일은 정상적으로 열렸지만 해당 정보가 존재하지 않는 경우"와  
"파일 자체를 열 수 없는 경우"가 모두 `false`로 처리되어 구분이 어려웠다.

예를 들어 학생 학번 중복 검사 함수에서는 다음과 같이 서로 다른 상태를 구분할 필요가 있었다.

- 중복이 있는 경우
- 중복은 없지만 파일은 정상적으로 읽은 경우
- 현재 선생님 정보가 없거나 파일을 열 수 없는 경우

그래서 단순히 `true / false` 대신 `int`를 반환하도록 구현했다.  
예를 들어 `1`은 중복 있음, `0`은 중복 없음, `-1`은 파일 열기 실패나 교사 정보 없음처럼  
각 상황을 다르게 구분해 처리할 수 있도록 만들었다.

이렇게 반환값을 세분화하니 단순히 "실패했다"는 것만 알 수 있는 것이 아니라,  
어떤 이유로 실패했는지에 따라 서로 다른 메시지를 출력하고 로직을 분기할 수 있어서  
예외 상황을 더 명확하게 처리할 수 있었다.

## 느낀점

이번 미니 프로젝트를 만들면서 Qt에서 여러 위젯을 연결해 프로그램 흐름을 구성하는 방법을 연습할 수 있었다.  
특히 단순히 화면만 구현하는 것이 아니라, 로그인 정보와 데이터 저장 구조까지 함께 생각해야 한다는 점이 어렵기도 했다.

이번에는 텍스트 파일 기반으로 구현했지만,  
학생 수나 데이터가 더 많아진다면 파일을 직접 관리하는 방식보다는 DB를 활용하는 것이 더 효율적일 것 같다는 생각이 들었다.

또 이번 프로젝트에서는 대부분의 기능을 하나의 `widget.cpp` 안에서 처리했는데,  
기능이 많아질수록 코드가 길어지고 읽기 어려워진다는 점도 느꼈다.  
실습평가 때는 다양한 기능을 하나의 클래스에 몰아넣기보다, 역할별 클래스로 분리해 구현하는 것을 목표로 하고 있다.
